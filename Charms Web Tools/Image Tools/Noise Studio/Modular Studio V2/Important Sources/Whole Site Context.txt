## Critical Development Rule

Logging is now a first-class system in this app.

- Modular Studio V2 now includes a top-level `Logs` section that surfaces site-wide process cards for startup, navigation, Library loading, import/export, save flows, sync work, Composite activity, Stitch activity, 3D activity, and Settings/maintenance work.
- Future updates must explicitly consider where new work should emit logs for the Logs tab.
- Do not add major background processing, save paths, import/export flows, synchronization checks, render stages, or recovery/error paths without wiring them into the logging system.
- The logging engine and UI are part of the app architecture now, not optional debug-only extras.
- If a future change introduces a new workflow and the code does not also update relevant logging points, that change should be treated as incomplete.

## 1. Product Model

Modular Studio V2 is now a single-page app with seven top-level sections, in current UI order:

1. `Editor`
2. `Composite`
3. `Library`
4. `Stitch`
5. `3D`
6. `Settings`
7. `Logs`

All seven sections live inside the same page and same JS runtime.

Important historical note:

- The old separate-window Library workflow was retired from the main path because it was unreliable on GitHub Pages even when it seemed fine on local hosting.
- `library.html` is now only a redirect to `index.html?section=library`.

Current section responsibilities:

- `Editor`
  - The original modular image-processing engine.
  - Uses the registry-driven layer stack and the WebGL pipeline.
  - Owns normal state-file save/load for studio documents.
  - Newly saved Editor JSON files are now self-contained: they embed both the original `source` image and an optional `preview` object containing the latest rendered PNG plus dimensions/timestamp.

- `Library`
  - Integrated project browser backed by IndexedDB.
  - Can hold mixed project types from `Editor`, `Composite`, `Stitch`, and `3D`, plus a separate asset catalog for images and reusable 3D models.
  - The Library page now has its own local `Library` and `Assets` page tabs.
  - Owns Library/Asset import-export, tag catalog, selection workflows, secure bundle save/load, asset-folder exchange, preview gallery, and detail inspection.
  - Quietly backfills older saved Editor Library records that are missing embedded `preview` data by re-rendering them with the Editor engine and logging that repair work in `Logs`.

- `Stitch`
  - A second engine with its own document model and UI.
  - Handles overlap-first multi-image stitching with candidate generation and manual correction.
  - Saves into the same Library as Editor, Composite, and 3D projects.

- `3D`
  - Browser-based 3D scene workspace built on three.js.
  - Supports embedded `.glb` / self-contained `.gltf` models, image planes, primitive meshes, flat/extruded text, 2D shapes, editable point / spot / directional lights, world light / HDRI lighting, and path tracing.
  - Saves full 3D scenes into the same Library as Editor, Composite, and Stitch, including embedded asset data, materials, textures, camera state, render settings, and preview image.
  - Also has a bottom content drawer that mirrors the Library `Assets` page and can drag saved assets directly into the current scene.

- `Composite`
  - A separate rendering tab (`src/composite/`) allowing users to lay out multiple saved Editor projects, imported images, text objects, and simple shape objects into one scene.
  - Features its own 2D rendering engine, world-space/infinite-canvas style stage, local interaction session for drag/pan/zoom work, and robust draggable UI bounds-based exporter.
  - The current Composite sidebar has `Layers`, `Transform`, `Blend`, and `Export` views.
  - Supports self-contained Composite JSON save/load, Library save/load, and a bridge flow for updating the original Editor Library project when Composite was opened from that project.
  - Integrated with the Library and has a "Pixel Match" logic allowing exports to render perfectly at the explicit resolution of bounded items.

- `Settings`
  - Top-level app settings surface for theme, worker limits, Library defaults/maintenance, Editor defaults, Composite viewport preferences, Stitch defaults/diagnostics, 3D preferences/defaults, and Logs behavior.
  - Owns app-level settings export/import plus maintenance actions that are intentionally separate from normal project payloads.

- `Logs`
  - Process-monitor section that shows live card-based logs for app bootstrap, workspace navigation, Library sync/import/export/save activity, Editor file/export flows, Composite work, Stitch work, 3D work, and Settings/maintenance work.
  - Each process card can be cleared independently and downloaded as a text log.
  - Intended as both user-facing transparency and a developer/debugging surface.

## 4. Shell and Section Routing

Top-level section switching is controlled by `state.ui.activeSection`.

Allowed values:

- `'editor'`
- `'composite'`
- `'library'`
- `'stitch'`
- `'3d'`
- `'settings'`
- `'logs'`

`src/main.js` reads `?section=` on startup:

- `?section=composite` -> Composite
- `?section=library` -> Library
- `?section=stitch` -> Stitch
- `?section=3d` -> 3D
- `?section=settings` -> Settings
- `?section=logs` -> Logs
- anything else -> Editor

`actions.setActiveSection(section)` syncs the URL with `history.replaceState(...)`.

Important behavior:

- Switching `Editor <-> Stitch` does not trigger save/discard prompts.
- Studio and Stitch keep separate in-memory drafts at the same time.
- Replacement prompts happen only when loading/replacing a project inside one engine.
- Hidden-section rendering is now intentionally deferred:
  - hidden Editor preview rerenders are postponed until returning to `Editor`
  - hidden Stitch canvas rerenders are postponed until returning to `Stitch`
  - the live 3D viewport pauses while `3D` is unfocused
  - background 3D PNG renders continue even while the 3D section is hidden
  - Library page refreshes and 3D asset-drawer refreshes are deferred while those sections are hidden

Cleanup on non-Editor section switches:

- batch playback stops
- compare overlays close
- eyedropper target is cleared
- Editor batch UI is forced closed

Current Composite layer kinds:

- `editor-project`
  - stores an embedded Editor payload plus optional original Library-link metadata
- `image`
  - stores an embedded image asset payload directly in the Composite document
- `text`
  - stores text content plus font family / font size / color
- `square`
  - stores a generated 2x2 square source plus color
- `circle`
  - stores a generated 2x2 circle source plus color
- `triangle`
  - stores a generated 2x2 triangle source plus color

Important current Composite behavior:

- Composite can export by automatic document bounds or by explicit draggable custom bounds.
- Switching Export to `custom` now seeds the bounds from the visible viewport/document intersection when possible instead of always jumping to the full document bounds.
- Custom export resolution stays aspect-locked to the current custom bounds, and pixel-match is only valid for visible layers fully contained inside those bounds.
- Composite uses runtime-loaded layer dimensions for selection handles and pixel-match/export calculations so Editor-backed preview layers do not rely on stale embedded size metadata.
- Current single-selection transform modes are `center-uniform`, `anchor-uniform`, and `anchor-stretch` where the selected layer type allows stretching.
- Composite also has a local multi-selection path: right-drag on empty stage builds a marquee selection, grouped bounds appear around the selected items, and dragging a selected item or the grouped handles moves/scales the selection together.
- Multi-selection and grouped handle mode are current UI-only state; the persisted payload still stores only `selection.layerId`.
- Composite supports local text and simple shape objects directly on the canvas in addition to Editor/image layers.
- Composite checker-background and wheel-zoom preferences now live in the top-level `Settings` section under the `Composite` category instead of a Composite-local settings panel.
- The quick `Use View As Export` action lives in the Composite `Export` panel because it is an export workflow action, not an app-level setting.
- Composite pan/zoom view changes do not trigger Library autosave by themselves.
- Composite save/load uses the `composite-document` payload shape.
- `preview` is optional durable metadata used by Library/local save flows.

Current saved 3D scene item types:

- `model`
  - stores embedded `asset.dataUrl`
  - supports `format: 'glb' | 'gltf'`
  - supports per-item `material`
    - `preset: 'original' | 'matte' | 'metal' | 'glass' | 'emissive'`
    - color / roughness / metalness / opacity
    - emissive settings: emissiveColor / emissiveIntensity
    - glass settings: ior / transmission / thickness / attenuationColor / attenuationDistance
    - optional embedded texture with repeat / offset / rotation
- `image-plane`
  - stores embedded image `asset.dataUrl`
  - stores original `asset.width` and `asset.height`
  - initial plane geometry preserves source aspect ratio
- `primitive`
  - stores `asset.format: 'primitive'`
  - stores `asset.primitiveType: 'cube' | 'sphere' | 'cone' | 'cylinder'`
  - supports the same saved material/texture settings as normal model items
- `text`
  - stores `text.content`
  - stores `text.mode: 'flat' | 'extruded'`
  - stores `text.fontSource.type: 'system' | 'upload'`
  - stores either `text.fontSource.family` or `text.fontSource.assetId`
  - stores `text.color`, `text.opacity`, emissive-style `text.glow`, sparse `text.characterOverrides`, extrude/bevel settings, and optional `text.attachment`
- `shape-2d`
  - stores `shape2d.type: 'square' | 'circle'`
  - stores `shape2d.color`, `shape2d.opacity`, and emissive-style `shape2d.glow`
- `light`
  - stores `lightType: 'directional' | 'point' | 'spot'`
  - stores color/intensity/shadow parameters
  - directional and spot lights can optionally store `targetItemId` to keep aiming at a chosen scene item

Important saved 3D asset buckets:

- `assets.fonts[]`
  - embedded uploaded `.ttf` / `.otf` / `.woff` / `.woff2`
  - flat text can use either uploaded fonts or browser/system families
  - extruded text must use an uploaded font asset
- `assets.hdris[]`
  - embedded uploaded `.hdr`
  - used by `scene.worldLight.mode === 'hdri'`

Important runtime-only 3D behavior:

- `renderJob` is store state, not durable project payload.
- Background render jobs continue while the user switches between Editor / Library / Stitch / 3D.
- While a background 3D render is active, the 3D viewport and 3D controls are intentionally locked except for `Abort`, so the render snapshot cannot be mutated mid-sample.
- 3D render progress is pushed into local 3D UI first and only checkpoints shared state, so the rest of the app remains usable while a render is running.
- Camera navigation supports both `orbit` and `fly` modes.
- Fly mode uses viewport focus plus right-drag look and `W A S D Q E` movement.
- Right-drag fly-look remains the normal camera-look gesture; the viewport object context menu only opens on right-click when the clicked object is already selected.
- View navigation also supports a canvas-style mode with:
  - `view.navigationMode: 'canvas'`
  - `view.projection: 'perspective' | 'orthographic'`
  - locked view presets `front | back | left | right | top | bottom | current`
  - rotation frozen while panning remains enabled
  - wheel behavior driven by `view.wheelMode`, with orthographic `Alt + Wheel` depth travel
- Orbit mode supports `F` to frame the currently selected object.
- Viewport render mode shortcuts now include keyboard `1` = edit (`raster`), `2` = path trace, and `3` = mesh.
- Plane-handle scaling can lock the two active axes together via `view.linkPlaneScale`.
- Transform snapping supports invisible-grid translation snapping plus rotation snapping via `view.snapTranslationStep` and `view.snapRotationDegrees`.
- Path tracing now has a warmup/loading overlay so the first compile/switch does not look like a frozen viewport.
- Path-tracing quality settings now persist in the 3D document, including bounce counts, glossy firefly filtering, and optional denoise parameters.
- `render.exportEngine` persists the last physically based export choice independently from preview `render.mode`.
- Background PNG renders now open through a single in-app modal for samples + width + height, and show explicit preparation / compile / encode messages before sampling starts.
- Background PNG renders can now optionally run a final built-in beauty-pass denoise step before the PNG is encoded.
- Edit view now forces unlit viewport materials and caps live viewport pixel ratio on high-DPI displays so object editing stays clearer and faster than the path-traced preview.
- Solid model / primitive items can now store persistent boolean subtraction snapshots captured from other objects, and `Reset Cuts` restores the original uncut target.

## 7. Stitch Engine Architecture

Stitch is a separate subsystem, not a layer mode inside the Studio engine.

### Current Stitch analysis goals

- Overlap-first, not collage-first.
- Candidate gallery first, then manual refinement.
- Split analysis backends:
  - screenshot-style inputs use the JS/Canvas matcher
  - photo-style inputs use the OpenCV.js/WASM worker
- Keep the existing Stitch placement/render/export model by materializing analysis results into rigid wrappers plus optional warp meshes.

Still important limitations:

- Mesh regularization is still heuristic, not a full global bundle/mesh optimizer.
- Photo stitching is stronger than the original rigid path, but it is not yet a dense panorama-grade solver.
- The screenshot backend and photo backend are intentionally different systems with different strengths.

### Stitch analysis pipeline

The current pipeline is split by backend.

`src/stitch/engine.js` now does:

1. normalize the document and prepare grayscale inputs
2. if `sceneMode === 'auto'`, run the lightweight classifier to choose screenshot vs photo
3. route screenshot sets to the JS analysis path
4. route photo sets to `src/stitch/opencv-worker.js`

The screenshot backend (`src/stitch/analysis.js`) still does:

1. downscale each input by default
2. gradient/edge-style preprocessing
3. screenshot-first overlap search for same-scale captures
4. translation/similarity matching and rigid candidate ranking

The photo backend (`src/stitch/opencv-worker.js`) now does:

1. grayscale prep at analysis resolution or full resolution
2. ORB feature extraction with AKAZE fallback / override support
3. BF KNN descriptor matching with Lowe-ratio and mutual filtering
4. RANSAC homography solving with overlap/inlier/reprojection rejection
5. graph traversal to compose image-to-anchor homographies
6. homography-derived `perspective` candidates
7. dense overlap support seeding across the shared region, not just sparse match inliers
8. a seam-focused two-image mesh profile that allocates more support near the overlap corridor
9. edge-aware optical-flow mesh refinement for `mesh` candidates, including a heavier two-stage forward/backward-validated flow pass for two-image photo seams
10. adaptive mesh sizing based on support density and residual disagreement
11. ranked candidate scoring with `rank`, `confidence`, `modelType`, and `blendMode`
12. manual fallback candidate if overlap is weak or missing

Important implementation detail:

- photo candidate traversal is currently cycle-guarded; without that, pair graphs can loop forever during anchor solving
- photo mesh refinement now supports two deformation-sharing modes:
  - `warpDistribution: 'balanced'`
    - spreads residual deformation across both images
  - `warpDistribution: 'anchored'`
    - keeps more deformation on the deeper/non-anchor side
- the photo mesh solver now seeds many more local support points from overlap detail/edge regions instead of relying only on the original inlier matches
- when exactly two photo inputs are being solved into a mesh, support seeding now deliberately favors the seam corridor instead of treating the whole overlap equally
- low-detail / blank-space regions are now deprioritized or skipped during dense support seeding so empty areas do not steer the mesh
- local mesh corrections are now thresholded twice:
  - implausibly large prepared-space optical-flow shifts are rejected
  - surviving world-space local residuals are capped to a small per-image warp budget
- two-image photo mesh refinement now uses a coarse-to-fine Lucas-Kanade pass plus forward-backward validation before residuals reach the mesh
- adaptive mesh density is now chosen per solved image from support count and residual disagreement, rather than always using one fixed grid size
- mesh residuals are intentionally regularized so unsupported outer edges stay closer to rectangular instead of letting a local seam pull the whole far edge apart
- that regularization is now less aggressive inside supported seam regions for two-image meshes, while unsupported outer regions stay more strongly pinned
- large local corrections are no longer rewarded just for being large; the mesh now prefers small believable cleanup corrections over aggressive stretches
- the Laplacian blend path now preserves combined source coverage alpha at overlap boundaries so narrow top/bottom seams do not fade out into background-colored lines
- automatic AKAZE fallback is now capped and non-fatal: Stitch keeps a usable ORB solve if the stronger retry is skipped or fails
- the Stitch settings and photo worker now recognize a `sift` feature-detector mode, and the vendored `src/vendor/opencv/opencv.js` bundle is now SIFT-enabled so explicit SIFT selection and auto escalation both work at runtime
- the repo includes `tools/generate_opencv_sift_config.py` plus `tools/build_opencv_sift.ps1` to regenerate that SIFT-enabled single-file OpenCV.js bundle
- the PowerShell build script also patches OpenCV's JS CMake input to remove the deprecated `DEMANGLE_SUPPORT` linker flag, which newer Emscripten toolchains reject during the final JS link step
- the custom OpenCV.js bundle exposes a promise-like `cv` during worker boot, so `src/stitch/opencv-worker.js` now explicitly swaps in the resolved runtime module before analysis begins; without that, ORB/AKAZE/SIFT appear missing or non-constructible in the browser worker even though the bundle contains them
- browser-side SIFT on this OpenCV.js build has a practical WASM ceiling well below many phone-photo full-res inputs, so Stitch now caps SIFT extraction to stable internal analysis sizes (roughly 2400px longest side, with smaller retries if needed), then maps those keypoints back into the original prepared-image coordinates before the rest of the photo pipeline runs

If the browser supports Workers:

- screenshot analysis runs through `src/stitch/worker.js`
- photo analysis runs through `src/stitch/opencv-worker.js`

If Worker or WASM support is missing:

- screenshot analysis can still fall back to same-thread JS analysis
- photo analysis currently requires the OpenCV worker path and will fail with a visible error instead of silently falling back

Important Stitch analysis setting:

- `settings.useFullResolutionAnalysis`
  - `true` by default for new Stitch drafts
  - when enabled, the prep step skips the normal downscale and analyzes the original image sizes
  - the screenshot search path also stays in original resolution instead of silently collapsing back to a tiny search proxy
  - this is especially useful for screenshots and UI captures, where downscaling can throw away the sharp edges and tiny features the matcher needs
  - it can be noticeably slower and use more memory

Current higher-level Stitch settings:

- `settings.sceneMode`
  - `'auto' | 'screenshot' | 'photo'`
- `settings.warpMode`
  - `'auto' | 'off' | 'perspective' | 'mesh'`

  - `'low' | 'medium' | 'high'`

  - `'auto' | 'alpha' | 'feather' | 'seam'`

  - `'auto' | 'orb' | 'akaze' | 'sift'`
- `settings.photoBackend`
  - currently `'opencv-wasm'`

Current new-document defaults are intentionally photo-heavy:

## 8. Project Adapter System

The Library save/load/import pipeline is now engine-aware.

This is handled in `src/main.js` via `createProjectAdapterRegistry(...)`.

Current adapters:

- `studio`
- `composite`
- `stitch`
- `3d`

Each adapter is responsible for:

- returning the engine's current draft from app state
- determining whether that draft is empty/saveable
- serializing a clean project payload
- validating imported payloads
- rendering a Library preview blob
- restoring a payload back into the correct engine
- providing summary metadata for the Library record

This adapter layer is what makes mixed Library storage possible without hard-coding Library behavior to the Studio engine.

## 9. Library Storage Model

The Library is backed by IndexedDB:

- DB name: `ModularStudioDB`
- object store: `LibraryProjects`

There is also a reserved metadata record:

```text
id: '__library_meta__'
kind: 'library-meta'
tags: [...]
```

That record stores the persistent tag catalog so tags can exist even if no current project is using them.

### Compare Source rule

For Studio Library projects:

- `hoverSource` is the embedded original source image.

For Stitch Library projects:

- `hoverSource` is the first uploaded input image.

This is intentional. It makes hover-compare still work in a multi-image project without inventing a composite "source".

### Compare Source rule for Composite

For Composite Library projects:

- `hoverSource` is intentionally `null`
- the Library card uses the saved Composite preview image instead of pretending one layer is the original source

### Source size sorting rule for Stitch

For Stitch Library projects:

- `sourceWidth/sourceHeight` track the first input image for compatibility and hover display
- `sourceAreaOverride` stores aggregate total input area across all inputs

The Library sort UI uses that override so Stitch projects sort meaningfully by original input size.

### Compare Source rule for 3D

For 3D Library projects:

- `hoverSource` is intentionally `null`
- the Library card uses the saved scene preview image instead of an original-source hover compare

## 11. Save / Load Behavior

### Save Editor state JSON

- Editor `Save` now captures the current rendered PNG and embeds it into `payload.preview`.
- The old Editor `Save Image` toggle was removed; Editor JSON saves are now intended to be self-contained by default.
- If there is no embeddable source image, the app still saves JSON but warns that source/render embedding was unavailable.

### Save Composite state JSON

- Composite `Save` now writes a self-contained `composite-document` JSON file.
- The save flow captures a fresh Composite preview before writing the JSON.
- If the current Composite draft came from a Library record, the local save flow also attempts to sync that active Composite Library record through the current overwrite/autosave path.
- Composite autosave for Library-linked drafts is delayed and commit-based; viewport-only pan/zoom changes are not supposed to trigger it.

### Save to Library

`actions.saveProjectToLibrary(...)` is now adapter-based.

Flow:

1. Determine which engine is being saved.
2. Ask that adapter for the current draft.
3. Refuse save if the draft is empty or not saveable.
4. Serialize a clean payload.
5. Fingerprint it with `makeProjectFingerprint(...)`.
   - Studio `preview` metadata is stripped from the fingerprint so duplicate detection compares editable document state, not rendered PNG bytes.
6. Reuse an exact existing Library record when `preferExisting` is requested.
7. If the draft originated from a Library record, prompt for overwrite-vs-new behavior.
8. Ask the adapter to render a preview blob and summary metadata.
9. For Studio projects, embed that freshly captured render into `payload.preview` before persistence while still storing the normal Library preview blob separately.
10. Save the resulting mixed-engine record into IndexedDB.
11. Preserve tags on overwrite.
12. If the saved project is an Editor project, upsert its derived Asset Library image record too.

### Load from Library

`actions.loadLibraryProject(payload, libraryId, libraryName)` now:

1. inspects the payload
2. chooses the correct adapter
3. runs the engine-specific replacement guard
4. restores the payload through that adapter
5. updates the correct active Library origin
6. switches to the correct top-level section

### Library project preview backfill

- `actions.getLibraryProjects()` first checks saved Editor Library records for missing `payload.preview.imageData`.
- Missing Studio previews are quietly re-rendered with the live Editor engine, written back into the IndexedDB project payload, and summarized in `Logs`.
- After the repair pass, the Editor engine restores the active live document so the backfill does not become the new working state.

### Asset Library synchronization

`actions.getLibraryAssets()` now returns dedicated asset records, not project records.

Important sync rules:

- Editor project saves populate the Asset Library with a derived image record linked to that project id.
- `importThreeDModelFiles(...)` and `importThreeDImageFiles(...)` quietly save reusable asset records after file-picker import.
- `addLibraryAssetToThreeDScene(assetId)` converts a saved asset record back into a live 3D scene item.
- 3D scene restore/import paths call `ensureLibraryAssetsFromThreeDItems(...)` so saved scene assets are backfilled into the Asset Library if needed.
- New/imported image assets generate `previewDataUrl` thumbnails on save, and older image assets missing previews are backfilled once per reload.
- `notifyLibraryChanged()` now coalesces refreshes, then updates both the Library page and the 3D content drawer.
- 3D Library scene restore now commits the `3D` section immediately, then backfills any missing reusable asset records asynchronously so the section switch feels instant.

### Active Library origin tracking

The app no longer tracks only one active Library project.

It now tracks separate origins for:

- `studio`
- `composite`
- `stitch`
- `3d`

That means:

- Editor can know which Library record it came from
- Composite can know which Library record it came from
- Stitch can know which Library record it came from
- switching sections does not destroy overwrite context

## 12. Replacement Guards

Project replacement checks are now engine-specific.

The helper:

- `ensureProjectCanBeReplaced(projectType, actionLabel)`

does this:

1. gets the correct adapter
2. checks if that engine's current draft is empty
3. checks whether the current draft already matches an exact Library record
4. if not, asks the user whether to save first
5. if the user chooses save, it routes through `actions.saveProjectToLibrary(..., { projectType })`

This is why:

- loading a Studio state file does not look at the Stitch draft
- loading a Stitch Library project does not look at the Studio draft
- simple tab switching still does not prompt

4. validates the payload
5. renders a preview blob with the correct engine
6. stores a mixed-engine record in IndexedDB

This means Editor, Composite, Stitch, and 3D projects can coexist in:

- single Library records
- bundle exports
- secure bundle exports
- ZIP exports

It also means:

- Asset-only bundles can load without touching the Library project cards
- combined bundles can restore both project records and asset records together
- folder imports can reconstruct Asset Library records without going through JSON upload

## 22. Worker hardening and transferable request prep

The shared worker runtime has now been hardened beyond the first bootstrap pass.

### New broker behavior

`src/workers/runtime.js` now supports:

- per-attempt worker request materialization through `options.createRequest`
- active-task cancellation that restarts the worker process for that domain instead of only posting `cancel`
- section/domain cancellation through the existing broker API, now used by the UI on workspace switches
- replay-safe worker request rebuilding, which matters for file-based tasks that transfer `ArrayBuffer` payloads

Important runtime rule:

- if a worker-backed task may transfer ownership of buffers, the task should prepare those buffers through `createRequest` so retries can rebuild them from the original source payload instead of reusing detached buffers

### Worker file payload helper

New file:

- `src/workers/filePayload.js`

This helper centralizes:

- converting `File` / `Blob` inputs into worker-safe `{ name, type, size, lastModified, buffer }` entries
- building transfer lists for worker dispatch
- checking cancellation while those request payloads are being prepared on the main thread
- reviving transferred entries back into `File`-like objects inside worker task handlers

Current use:

- Studio state-file parse
- Stitch uploaded image preparation
- 3D model/image/font/HDRI import preparation

This is an input-side optimization/hardening step only.

- worker task outputs for these flows still often resolve to `dataUrl`-based payloads because the current document/library formats still expect them
- future performance work can reduce output-side `dataUrl` traffic, but this pass mainly makes request dispatch safer and cheaper to retry

### Section-scoped worker ownership now enforced by the UI

`src/main.js` now cancels section-scoped worker jobs when the user leaves:

- `section:editor`
- `section:stitch`
- `section:3d`

Current worker-backed UI tasks now opt into scoped replacement:

- Studio state-file parsing uses:
  - `scope: section:editor`
  - `replaceKey: section:editor:read-studio-state-file`
- Stitch file preparation uses:
  - `scope: section:stitch`
  - `replaceKey: section:stitch:prepare-input-files`
- 3D file preparation uses:
  - `scope: section:3d`
  - task-specific replace keys for model/image/font/HDRI imports

Meaning:

- switching away from a workspace should stop its in-flight worker prep instead of letting it continue quietly in the background
- repeating the same import/open action should replace the previous in-flight prep job instead of stacking duplicate work

### Current reliability assumptions

- domain workers are still treated as stateless task executors
- restarting a worker on active cancellation is currently safe for `app-library`, `editor`, `stitch`, and `three`
- if a future worker starts holding expensive in-memory state or WASM warm state, cancellation strategy should be re-evaluated per domain before changing this default

### Current main-thread-only boundaries remain

This hardening pass still does **not** move:

- Studio live rendering/export/preview capture
- Stitch OpenCV photo worker onto the shared broker
- 3D scene hydration, viewport rendering, or path tracing
- IndexedDB ownership

## 23. Shared worker references and perf observer guard

Two small but important follow-up fixes landed after the worker hardening pass.

### Shared broker references

`backgroundTasks` and `workerCapabilities` now live at module scope in `src/main.js`.

Reason:

- some shared Library/helper functions are also defined at module scope
- those helpers can run during Library restore/backfill flows for Editor/Stitch/3D projects
- keeping the worker broker reference only inside the `DOMContentLoaded` closure caused a real `ReferenceError` during certain Library-driven 3D restore paths

Rule going forward:

- any module-scope helper that may use the shared worker broker must only reference the module-scope `backgroundTasks`
- avoid shadowing those shared worker references inside the bootstrap closure

### Performance observer guard

`src/perf/bootstrapMetrics.js` now checks `PerformanceObserver.supportedEntryTypes` before registering a `longtask` observer.

Reason:

- Firefox/local browser runs may not support `longtask`
- calling `observe({ entryTypes: ['longtask'] })` in those environments can produce noisy console warnings even when the app still works

Current behavior:

- if `longtask` is supported, bootstrap metrics keep recording long-task entries
- if `longtask` is not supported, the observer is skipped quietly and the rest of the startup metrics still work

### Follow-up fix after first live Library WASM test
