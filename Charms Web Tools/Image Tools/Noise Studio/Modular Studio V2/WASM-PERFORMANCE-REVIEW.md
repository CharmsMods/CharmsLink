# WebAssembly (WASM) Performance Opportunities

An evaluation of computational bottlenecks across the Modular Studio V2 Editor, Library, and 3D functionality. This review isolates operations currently processed in vanilla Javascript that would yield massive performance improvements if translated into WebAssembly environments (SIMD/compiled C++ or Rust). Note: All image-stitching functionality was intentionally bypassed for this evaluation.

## 1. Scope Analysis & Histogram Generation (Editor Tab)

**The Bottleneck:**
Inside `src/analysis/scopes.js`, the visual tools `updateHistogram`, `updateVectorscope`, and `updateParade` evaluate raw image intensity matrices entirely via JavaScript. The JS engine is iterating over massive subsets of `analysisPixelBuffer` (often between 12,000 to 20,000 downsampled pixels).
- **Vectorscope & Parades:** For each pixel sampled, `updateVectorscope` calculates trigonometric translations (`Math.sin`, `Math.cos`), and generates rendering coordinates, utilizing `ctx.fillRect` natively in `for` loops. Generating parades makes up to **60,000 separate `ctx.fillRect` draw calls** across three color bands on the main UI thread.

**The WASM Solution:**
The sheer volume of Canvas 2D math generated by these pixels locks the browser event loop during live, high-framerate playback. By passing the flat `analysisPixelBuffer` pointer into a WASM module, the module can execute native SIMD math to build a single consolidated 8-bit visual buffer array (e.g., a resulting 256x256 image blob). Back in Javascript, you render this composite just once per frame via `ctx.putImageData`, drastically plummeting CPU usage.

## 2. Visual Diffing Engine (Editor Tab)

**The Bottleneck:**
Within the main pipeline (`src/engine/pipeline.js`), `drawChangedPixelsPreview()` handles the UI overlay that visualizes what pixels a layer effect changed. 
- It simultaneously loops over `thumbClampedBuffer` and `thumbClampedBufferAlt`. 
- Over 102,400 coordinate offsets (~409,600 elements), it runs Euclidean color distance equations (`Math.sqrt(...)`) and Luma computations in blocking JavaScript logic.

**The WASM Solution:**
Array differential mathematics is fundamentally what WebAssembly excels at due to linear memory tracking. Rewriting this diff calculation logic in AssemblyScript, Rust, or straightforward C++ and dropping it into WASM would take milliseconds and alleviate garbage-collection (GC) memory thrashing natively caused by complex loop variables in V8.

## 3. Constructive Solid Geometry & Mesh Intersections (3D Tab)

**The Bottleneck:**
`src/3d/engine.js` boots the `three-bvh-csg` module to perform real-time boolean actions (Difference, Union) against imported meshes. 
- Javascript CSG engines, while robust, calculate thousands of vertex intersections, split triangle faces dynamically, and retriangulate gaps using arrays.
- Even at moderate polygon counts (10k - 50k faces), running this computation directly natively in JS triggers a massive thread hang before updating the WebGL renderer.

**The WASM Solution:**
You can transition the boolean engine off `three-bvh-csg` and onto the **Manifold3D** library, which boasts an official WebAssembly wrapper. Since Manifold is written directly in C++ and optimized for rapid geometric evaluations, the mesh vertex layouts are evaluated at nearly native desktop speeds, allowing for highly complex architectural boolean intersections seamlessly.

## 4. Compressed Payload Decoders (3D Tab)

**The Bottleneck:**
The `src/3d/engine.js` script natively depends on `GLTFLoader` with no auxiliary compression layers enabled. High-fidelity 3D assets loaded off external CDNs or dropped in heavily punish RAM architectures as the `.gltf` buffers are decoded completely uncompressed inside the Javascript V8 boundary.

**The WASM Solution:**
Integrating `DRACOLoader` (Mesh Compression) and `KTX2Loader` (Basis Universal Texture Compression) natively interfaces with designated WASM decoders officially supported by Three.js. This permits gigantic textures to sit in GPU memory compactly, and allows massively complex scenes to hydrate off tiny wire payloads instantly due to WASM's binary-streaming optimizations.

## 5. Security Iterations & Blob Encoding (Library Tab)

**The Bottleneck:**
Exporting dense, offline JSON `library-bundles` pushes memory barriers for standard web constraints. When encrypting payload chunks (`src/ui/libraryPanel.js`), functions like `bytesToBase64` execute strict 32kb chunk allocations mapping bytes via `String.fromCharCode(...chunk)`.
- Exporting a heavily stacked library JSON or handling gigabytes of assets locally pushes Javascript array allocators exceptionally hard during string transpositions, heavily ballooning memory requirements.

**The WASM Solution:**
Exporting binary-safe Base64 streams in pure C++/WASM processes allows memory arrays to be converted virtually instantaneously without allocating thousands of disposable strings for the JS garbage collector to clean up. Implementing a fast WASM string encoding step prior to packaging the final Library downloads ensures that exporting 200MB+ studio library files won't fatally crash iOS Safari or low RAM Chrome sessions.
