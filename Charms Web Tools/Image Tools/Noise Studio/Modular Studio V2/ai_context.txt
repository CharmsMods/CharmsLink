# Modular Studio V2 - Architecture and Context Guide

This file is the AI handoff document for Modular Studio V2. It should let a new model understand the codebase quickly without having to rediscover the architecture from scratch.

The project is a browser-based image editor with a custom WebGL pipeline, a registry-driven layer system, and a same-origin Library workflow that stores rendered projects in IndexedDB and can be used from a separate window or tab.

## 1. High-Level Product Model

There are two main browser surfaces:

1. `index.html` -> the main editor.
2. `library.html` -> the Library window/tab.

They are separate pages, but they are intentionally part of the same application. They share:

- the same origin
- the same IndexedDB database
- a cross-window command channel

The editor owns rendering, image processing, and the authoritative live document state.

The Library owns:

- browsing saved projects
- exporting/importing Library JSON bundles
- project inspection UI
- initiating "load into editor" and "render uploaded Library JSON files" commands

Important design intent:

- The Library is not supposed to be a clone of editor state.
- The Library reads saved records from IndexedDB.
- The editor performs rendering work and writes Library records into IndexedDB.
- The two pages communicate mostly for commands, progress, and refresh notifications.

## 2. Directory Structure and Responsibilities

### Root files

- `index.html`
  - Minimal shell for the main editor.
  - Mounts `<div id="app"></div>`.
  - Loads `src/main.js` as an ES module.

- `library.html`
  - Physical page for the Library UI.
  - Contains inline page-specific CSS.
  - Loads bundled `src/jszip.min.js`.
  - Loads `src/ui/library.js` as an ES module.

- `style.css`
  - Shared editor styling.

- `ai_context.txt`
  - This document.

- `LEGACY/`
  - Older V1-era files and shader JSON definitions.
  - Usually not part of the current execution path.
  - Useful for historical reference only.

### `src/`

- `src/main.js`
  - Main editor entry point.
  - Creates the store and initial document state.
  - Loads the registry.
  - Creates the WebGL engine.
  - Creates the workspace UI.
  - Wires editor actions to state mutations, engine calls, exports, imports, and Library behavior.
  - Owns the editor-side IndexedDB helpers and the editor-side Library message handling.

- `src/ui/workspaces.js`
  - Main DOM renderer for the editor.
  - Maps app state to markup.
  - Defines the toolbar and sidebar structure.
  - Delegates button clicks and form events into `actions.*`.

- `src/ui/library.js`
  - UI logic for `library.html`.
  - Reads Library records from IndexedDB.
  - Renders grid/fullscreen/detail views.
  - Sends commands to the editor via the shared channel.
  - Refreshes when the Library DB changes.

- `src/ui/previewViewport.js`
  - Editor preview viewport helpers.

- `src/io/documents.js`
  - JSON import/export helpers.
  - Validation entry point: `validateImportPayload(...)`.
  - File reading for saved document JSON.

- `src/io/crossWindowChannel.js`
  - Shared transport abstraction for editor <-> Library communication.
  - Wraps `BroadcastChannel`, same-origin `postMessage`, and `localStorage` storage events.
  - Prevents duplicate processing when the same message arrives through multiple transports.

- `src/state/store.js`
  - Lightweight state container used by the editor.

- `src/state/documentHelpers.js`
  - Helpers for view state, normalization, and document-level utilities.

- `src/registry/`
  - Registry loading and validation.
  - `layerRegistry.json` is the canonical layer manifest.
  - `index.js` instantiates layer instances and validates registry structure.
  - `utilityPrograms.json` contains shared low-level shader/program metadata.

- `src/engine/`
  - Custom WebGL engine.
  - `pipeline.js` is the main execution path.
  - `bootstrap.js` and `executors/` support texture/program setup and specialized execution behavior.

- `src/analysis/scopes.js`
  - Analysis/scope helpers for editor previews.

- `src/jszip.min.js`
  - Bundled JSZip used by the Library ZIP export.

### `Shaders/`

- Physical GLSL fragment and vertex shader files.
- These are the assets consumed by the registry and engine.

## 3. Main Runtime Model

### 3.1 Editor boot flow

When `index.html` loads:

1. `src/main.js` runs on `DOMContentLoaded`.
2. The layer registry is loaded.
3. A cross-window Library channel is created.
4. The editor store is created from `createInitialState()`.
5. `NoiseStudioEngine` is created.
6. The workspace UI is created with `createWorkspaceUI(...)`.
7. The engine is initialized on the main canvas.
8. Store subscriptions drive DOM rerendering and WebGL rerender requests.

### 3.2 Main editor state shape

The store state has several top-level zones:

- `state.document`
  - Persistent, saveable editor data.
  - This is the state shape that matters for document import/export and engine rendering.

- `state.ui`
  - Ephemeral UI flags that should not necessarily be persisted.

- `state.notice`
  - Current toast/message shown to the user.

- `state.eyedropperTarget`
  - Tracks active eyedropper mode and target control.

The most important part is `state.document`, which includes:

- `version`
- `kind`
- `mode`
- `workspace`
- `source`
- `palette`
- `layerStack`
- `selection`
- `view`
- `export`
- `batch`

### 3.3 Document source model

`document.source` stores the currently loaded image metadata:

- `name`
- `type`
- `imageData`
- `width`
- `height`

`imageData` is especially important because the Library system depends on it:

- saving to the Library requires `source.imageData`
- loading Library payloads into the editor restores the image from `source.imageData`
- uploaded Library JSON files must contain an embedded image if the editor is expected to render thumbnails from them

## 4. Render and Update Flow

The editor follows a fairly standard pattern:

1. A user action calls an `actions.*` method.
2. The action mutates store state through `updateDocument(...)`, `updateInstance(...)`, or `store.setState(...)`.
3. The store subscriber rerenders the DOM when needed.
4. If `meta.render` is true and the engine has an image, the engine is asked to render the current `document`.

Important rule:

- UI-only flags belong in `state.ui`.
- Anything that must affect engine rendering or be part of saved documents belongs in `state.document`.

## 5. Registry-Driven Layer System

Layers are defined primarily through `src/registry/layerRegistry.json`.

Each layer entry describes:

- metadata (`layerId`, label, group)
- executor mode
- default params
- UI controls
- shader bindings
- optional mask support

`src/registry/index.js` validates and loads the registry and is strict about accepted values. New layers must stay within the allowed validation sets unless the registry system itself is extended.

### 5.1 Layer instances

Editor layer instances are not just raw registry entries. They are concrete items in `document.layerStack`.

The editor creates them through registry helpers and later normalizes them with functions like:

- `createLayerInstance(...)`
- `relabelInstance(...)`
- `reindexStack(...)`

`reindexStack(...)` is important because it:

- rebuilds params from registry defaults plus saved overrides
- normalizes `enabled` / `visible`
- drops invalid or unknown layers
- reassigns labels/indexing for sibling instances

## 6. Engine Architecture

The WebGL engine is implemented in `src/engine/pipeline.js` and related engine files.

Core traits:

- custom render pipeline, not a third-party image editor
- ping-pong render targets
- dynamic buffer allocation based on the document's current stack and size changes
- layer-by-layer execution through registry-defined and manual executors

The engine is responsible for:

- loading source images
- shader compilation/use
- main preview rendering
- image export to PNG blobs
- layer previews / scoped analysis helpers
- comparison views

Layers that need unusual behavior such as:

- custom multi-pass logic
- size changes
- direct alpha manipulation
- special sub-previews

usually require explicit engine-side support instead of only registry changes.

## 7. Editor UI and Action Routing

`src/ui/workspaces.js` is the main editor UI composition layer.

It is responsible for:

- toolbar layout
- sidebar content
- control rendering
- event delegation to editor actions

Toolbar actions are routed through `data-action` values into `actions.*` methods in `src/main.js`.

Common toolbar mappings include:

- `save-state` -> `actions.saveState()`
- `save-to-library` -> `actions.saveProjectToLibrary()`
- `export-current` -> `actions.exportCurrent()`
- `compare-open` -> `actions.openCompare()`
- `new-project` -> `actions.newProject()`
- `open-library` -> `actions.openLibrary()`

## 8. Library System Overview

The Library is a persistent project browser backed by IndexedDB.

Conceptually, a Library entry contains:

- a rendered preview PNG blob
- the saved document payload
- user-facing name metadata
- timestamp/id metadata

The editor is the writer/renderer.
The Library page is the browser/inspector.

### 8.1 IndexedDB schema

Database:

- `ModularStudioDB`

Object store:

- `LibraryProjects`

Key path:

- `id`

Current record shape:

```js
{
  id: String,
  timestamp: Number,
  name: String,
  blob: Blob,
  payload: Object
}
```

### 8.2 IndexedDB helpers

Editor-side helpers live in `src/main.js`:

- `initDB()`
- `saveToLibraryDB(project)`
- `deleteFromLibraryDB(id)`
- `clearAllFromLibraryDB()`
- `getAllFromLibraryDB()`

Library-side helpers are duplicated inline in `src/ui/library.js` because that file runs in a separate page context.

This duplication is intentional and acceptable for now.

### 8.3 Save behavior and deduplication

`actions.saveProjectToLibrary(nameOverride, forceNew)` is the core save path.

It:

1. Verifies that there is a loaded source image with `source.imageData`.
2. Builds a payload containing:
   - `version`
   - `kind`
   - `workspace`
   - `export`
   - `source`
   - `layerStack`
3. Computes a fingerprint using `makeProjectFingerprint(payload)`.
4. Loads existing Library entries.
5. Deletes exact fingerprint duplicates found elsewhere in the DB.
6. Short-circuits if the active Library record already matches exactly.
7. Prompts the user for overwrite-vs-new naming behavior when the current document came from the Library.
8. Renders a fresh preview PNG via `engine.exportPngBlob(...)`.
9. Stores the result in IndexedDB.
10. Updates `activeLibraryId` and `activeLibraryName`.
11. Emits `LIBRARY_DB_UPDATED` so the Library UI can refresh.

Important nuance:

- Deduplication is global across the Library store, not just against the currently loaded record.

### 8.4 Tracking the loaded Library origin

The editor tracks the current Library origin with:

- `activeLibraryId`
- `activeLibraryName`

These are used to decide whether a save should:

- overwrite the original Library entry
- or create a new one

`newProject()` resets them.

### 8.5 Auto-save hooks related to the Library

Several actions automatically save into the Library:

- `openImageFile(file)`
  - After loading an image, the editor auto-saves to the Library.

- `openStateFile(file)`
  - After loading a JSON document with an embedded image, the editor auto-saves to the Library.

- `LOAD_PROJECT` handling
  - Before the editor loads a project requested by the Library, it auto-saves the currently loaded project if one exists.

This makes the Library function partly like a persistent working history.

## 9. Library Page Behavior

`library.html` + `src/ui/library.js` implement a separate UI shell for browsing the Library.

Primary behaviors:

- read all projects from IndexedDB
- render a project grid
- show fullscreen browsing mode
- show a project detail modal
- allow per-entry delete
- allow full-Library export
- allow ZIP export of all JSON payloads
- allow upload of JSON files to be rendered by the editor and saved into the Library
- allow "Load in Editor" from the detail modal

### 9.1 Library visual model

The Library uses a black/white, utility-style visual language:

- dark background
- white borders
- square-ish controls
- mono labels for technical metadata

The Library page has its own CSS embedded directly in `library.html`.

### 9.2 Library views

Main Library UI states include:

- grid view
- fullscreen click-to-advance series view
- project detail overlay
- loading overlay during editor-side batch rendering
- modal overlay for confirmation/alerts

### 9.3 Project detail modal

The detail modal is especially useful for AI understanding because it reflects how Library payloads are interpreted.

It shows:

- preview image
- hover source image
- project name
- layer count / active layer count
- source metadata
- filtered layer param cards

The filtering logic depends on `LAYER_DEFAULTS`, which comes from the editor at runtime. Parameters that still match registry defaults are hidden from the modal to keep the summary focused on changed values.

## 10. Editor <-> Library Communication

### 10.1 Current transport model

The editor and Library communicate through `src/io/crossWindowChannel.js`.

This replaced the older mental model of "use BroadcastChannel one way and postMessage the other way."

The channel abstraction intentionally sends each logical message through multiple possible same-origin routes:

- `BroadcastChannel`
- direct window `postMessage`
- `localStorage` storage events

This is a reliability feature, not redundancy by accident.

### 10.2 Why the transport is layered

The multi-window workflow historically appeared to work on local hosting but break on GitHub Pages. The likely issue was not IndexedDB itself. The real fragility was the command/refresh bridge between the Library and editor windows.

The current layered transport is meant to survive more real-world cases:

- editor opens the Library as a popup/window
- user reuses an existing named Library window
- Library is opened independently in another same-origin tab
- one transport path is unavailable or stale
- the editor reloads while the Library stays open

If the system ever regresses again on GitHub Pages but still works locally, investigate transport assumptions before redesigning the Library as an overlay.

### 10.3 `createCrossWindowChannel(...)`

`createCrossWindowChannel(channelName, options)` currently does the following:

- creates a per-window sender id
- creates a logical storage key of the form `ModularStudioChannel:${channelName}`
- optionally creates a `BroadcastChannel`
- listens for same-origin `message` events
- listens for `storage` events
- normalizes outgoing messages with metadata
- deduplicates repeated arrivals by message id

Current normalized metadata fields:

- `_channel`
- `_senderId`
- `_mid`
- `_sentAt`

Message deduplication matters because the same logical message can arrive through:

- broadcast
- direct `postMessage`
- `storage`

and should only be handled once.

Compatibility note:

- The channel accepts messages without `_channel` as a backward-compatibility measure so a fresh window can still interact with a stale same-origin tab running an older build until both are refreshed.

### 10.4 Direct window targets

The shared channel still supports direct target windows where available:

- Editor side:
  - `resolveTargetWindow: () => window.libraryWindow`

- Library side:
  - `resolveTargetWindow: () => window.opener`

This means a popup relationship is still useful when it exists, but it is no longer the only expected communication path.

### 10.5 Important message types

Current message types in active use:

- `REQ_LAYER_DEFAULTS`
  - Library asks the editor for registry defaults.

- `RES_LAYER_DEFAULTS`
  - Editor sends a `layerId -> defaults` map back.

- `RENDER_LIBRARY_FILES`
  - Library uploads one or more JSON payloads and asks the editor to render/store them.

- `LOAD_PROJECT`
  - Library asks the editor to load a selected project payload.

- `START_RENDER`
  - Editor tells the Library to show the loading overlay.

- `UPDATE_PROGRESS`
  - Editor updates Library progress state during batch processing.

- `LIBRARY_DB_UPDATED`
  - Editor tells the Library to reload from IndexedDB.

- `EDITOR_READY`
  - Editor announces that it is live and ready so the Library can resync.

## 11. How the Library Uses IndexedDB vs Messaging

This distinction is important.

### IndexedDB is used for:

- persistent saved Library entries
- Library grid data
- project preview blobs
- payload storage
- cross-page shared persistence

### Messaging is used for:

- requesting layer defaults
- asking the editor to render uploaded JSON files
- asking the editor to load a selected project
- showing progress during editor-side batch rendering
- telling the Library to refresh after DB changes
- announcing editor readiness

Do not confuse these responsibilities.

If saves appear to "not show up in the Library," the problem might be:

- save did not actually write to IndexedDB
- the Library did not refresh its DB view
- the cross-window signal failed

Those are different failure classes.

## 12. `actions.processLibraryPayloads(...)`

This action is important because it is where uploaded Library JSON files become actual Library entries.

Current flow:

1. The Library sends `RENDER_LIBRARY_FILES` with JSON strings and filenames.
2. The editor receives the message.
3. The editor sends `START_RENDER`.
4. For each payload:
   - parse JSON
   - validate payload
   - require embedded `source.imageData`
   - rebuild/normalize layer stack
   - load the source image into the engine
   - render a preview blob with `engine.exportPngBlob(...)`
   - save a new Library record into IndexedDB
   - send `UPDATE_PROGRESS`
5. Restore the editor's original source image if possible.
6. Send `LIBRARY_DB_UPDATED`.

Important recent change:

- This path no longer requires `window.libraryWindow` to exist.
- It can work even if the Library is not the editor-opened popup, as long as both pages are same-origin and the channel can reach the editor.

## 13. Opening the Library Window

`actions.openLibrary()` currently:

1. Reuses `window.libraryWindow` if it exists and is still open.
2. Otherwise resolves `library.html` against `window.location.href`.
3. Opens the page with named target `ModularStudioLibrary`.
4. Shows a warning notice if the popup was blocked.
5. Emits `EDITOR_READY` so the Library can resync immediately.

Important:

- The current architecture does not use `document.write`.
- There is no `isLibraryInitialized` bootstrap flag anymore.
- The Library is a real file, not dynamically generated HTML.

## 14. Library Resync Behavior

`src/ui/library.js` refreshes proactively in multiple situations:

- initial page load
- when it receives `LIBRARY_DB_UPDATED`
- when it receives `RES_LAYER_DEFAULTS`
- when it receives `EDITOR_READY`
- when the window gains focus
- when the page becomes visible again

This is intentional and should not be "simplified" without understanding why. It is part of the reliability strategy for multi-window use.

## 15. File Import / Export Formats

### 15.1 Standard editor document JSON

Handled through `src/io/documents.js` and editor save/open actions.

This represents the document state and may or may not include embedded image data depending on save settings.

### 15.2 Library full-project export

The Library "Save Full Project" action exports the entire Library as a JSON array.

Each entry looks like:

```json
{
  "_libraryName": "example-name",
  "version": 2,
  "kind": "document",
  "...": "payload fields"
}
```

`_libraryName` is used when re-importing into the Library so filenames can be restored.

### 15.3 Library ZIP export

The Library "Export All JSONs" action:

- creates a ZIP with JSZip
- writes one `.mns.json` file per Library entry
- ensures `{ version: 2, kind: 'document', ...payload }` shape
- preserves filenames where possible

The ZIP downloads as `modular_studio_exports.zip`.

## 16. Hosting and Deployment Notes

The editor and Library are designed for same-origin deployment.

This is required for:

- shared IndexedDB access
- safe same-origin `postMessage`
- `localStorage` event-based fallback transport

GitHub Pages should be a valid deployment target as long as:

- `index.html` and `library.html` are both published
- their relative asset paths remain valid
- both pages resolve under the same Pages origin

If the app works on localhost but not on GitHub Pages, likely suspects include:

- popup blocking or popup policy differences
- a stale open tab still running older JS
- path resolution/deploy path mismatch
- one transport API behaving differently in that browser
- same-origin assumptions being broken by how the pages are opened

Do not assume the solution is "move the Library into an overlay" until transport and hosting assumptions have been tested first.

## 17. Common Gotchas

### 17.1 Library architecture is static-page based

The Library is a dedicated file (`library.html`), not editor-injected HTML.

### 17.2 Fresh DB connections are intentional

Each DB helper opens IndexedDB fresh instead of sharing one long-lived connection. This reduces stale-connection edge cases between windows.

### 17.3 The Library detail modal depends on live registry defaults

If the detail modal starts showing too many params, check the `REQ_LAYER_DEFAULTS` / `RES_LAYER_DEFAULTS` handshake first.

### 17.4 Not every cross-window issue is a DB issue

The Library can fail to refresh even when the DB write succeeded.

### 17.5 Not every importable JSON can be rendered

`actions.processLibraryPayloads(...)` expects embedded source image data. Payloads without `source.imageData` cannot produce preview renders.

## 18. Guidance for Future AI Work

If you change any of these systems, update this file:

- editor/document state shape
- Library architecture
- IndexedDB schema
- cross-window transport
- file formats
- engine/render pipeline rules
- registry/layer architecture

Do not update this file for routine styling or one-off bug fixes unless they change the architectural truth future agents need to know.

If a future AI is debugging Library problems, it should trace issues in this order:

1. Did the editor write the expected record into IndexedDB?
2. Did the cross-window channel deliver the expected command/signal?
3. Did the Library refresh from IndexedDB afterward?
4. Is the issue local-only, Pages-only, or browser-specific?
5. Is the problem transport, persistence, or payload validity?
