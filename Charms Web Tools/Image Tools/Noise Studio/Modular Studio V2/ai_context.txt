# Modular Studio V2 - Current Architecture and AI Handoff Guide

This document describes the app as it works now.

It is meant to be handed to a future AI together with the site files so the AI can understand:

- what the product currently is
- which files own which responsibilities
- how save/load flows actually work
- where editor logic ends and Library/Stitch logic begin
- which old assumptions are now wrong

If the code and this file disagree, the code is authoritative and this file should be updated.

## 1. Product Model

Modular Studio V2 is now a single-page app with four top-level sections:

1. `Editor`
2. `Library`
3. `Stitch`
4. `3D`

All four sections live inside the same page and same JS runtime.

Important historical note:

- The old separate-window Library workflow was retired from the main path because it was unreliable on GitHub Pages even when it seemed fine on local hosting.
- `library.html` is now only a redirect to `index.html?section=library`.

Current section responsibilities:

- `Editor`
  - The original modular image-processing engine.
  - Uses the registry-driven layer stack and the WebGL pipeline.
  - Owns normal state-file save/load for studio documents.

- `Library`
  - Integrated project browser backed by IndexedDB.
  - Can hold mixed project types from `Editor`, `Stitch`, and `3D`.
  - Owns Library import/export, tag catalog, selection workflows, secure bundle save/load, preview gallery, and detail inspection.

- `Stitch`
  - A second engine with its own document model and UI.
  - Handles overlap-first multi-image stitching with candidate generation and manual correction.
  - Saves into the same Library as Editor projects.

- `3D`
  - Browser-based 3D scene workspace built on three.js.
  - Supports embedded `.glb` / self-contained `.gltf` models, image planes, primitive meshes, and editable point / spot / directional lights.
  - Saves full 3D scenes into the same Library as Editor and Stitch, including embedded asset data, materials, textures, camera state, render settings, and preview image.

## 4. Shell and Section Routing

Top-level section switching is controlled by `state.ui.activeSection`.

Allowed values:

- `'editor'`
- `'library'`
- `'stitch'`
- `'3d'`

`src/main.js` reads `?section=` on startup:

- `?section=library` -> Library
- `?section=stitch` -> Stitch
- `?section=3d` -> 3D
- anything else -> Editor

`actions.setActiveSection(section)` syncs the URL with `history.replaceState(...)`.

Important behavior:

- Switching `Editor <-> Stitch` does not trigger save/discard prompts.
- Studio and Stitch keep separate in-memory drafts at the same time.
- Replacement prompts happen only when loading/replacing a project inside one engine.

Cleanup on non-Editor section switches:

- batch playback stops
- compare overlays close
- eyedropper target is cleared
- Editor batch UI is forced closed

## 6. Studio Engine Architecture

The Editor side is still the original modular image-processing app:

- registry-driven layer stack
- custom WebGL rendering pipeline
- source image plus layer stack controls
- scopes and compare views
- batch folder processing

Normal flow:

1. A UI event calls an `actions.*` method.
2. `state.document` changes.
3. `createWorkspaceUI(...).render(state)` rerenders the shell.
4. If the update requests Studio rendering and the Studio engine has an image, the engine rerenders using `state.document`.

The addition of Stitch did not replace the Studio engine. It added a second engine beside it.

## 7. Stitch Engine Architecture

Stitch is a separate subsystem, not a layer mode inside the Studio engine.

### Current Stitch analysis goals

- Overlap-first, not collage-first.
- Candidate gallery first, then manual refinement.
- Split analysis backends:
  - screenshot-style inputs use the JS/Canvas matcher
  - photo-style inputs use the OpenCV.js/WASM worker
- Keep the existing Stitch placement/render/export model by materializing analysis results into rigid wrappers plus optional warp meshes.

Still important limitations:

- Mesh regularization is still heuristic, not a full global bundle/mesh optimizer.
- Photo stitching is stronger than the original rigid path, but it is not yet a dense panorama-grade solver.
- The screenshot backend and photo backend are intentionally different systems with different strengths.

### Stitch analysis pipeline

The current pipeline is split by backend.

`src/stitch/engine.js` now does:

1. normalize the document and prepare grayscale inputs
2. if `sceneMode === 'auto'`, run the lightweight classifier to choose screenshot vs photo
3. route screenshot sets to the JS analysis path
4. route photo sets to `src/stitch/opencv-worker.js`

The screenshot backend (`src/stitch/analysis.js`) still does:

1. downscale each input by default
2. gradient/edge-style preprocessing
3. screenshot-first overlap search for same-scale captures
4. translation/similarity matching and rigid candidate ranking

The photo backend (`src/stitch/opencv-worker.js`) now does:

1. grayscale prep at analysis resolution or full resolution
2. ORB feature extraction with AKAZE fallback / override support
3. BF KNN descriptor matching with Lowe-ratio and mutual filtering
4. RANSAC homography solving with overlap/inlier/reprojection rejection
5. graph traversal to compose image-to-anchor homographies
6. homography-derived `perspective` candidates
7. dense overlap support seeding across the shared region, not just sparse match inliers
8. a seam-focused two-image mesh profile that allocates more support near the overlap corridor
9. edge-aware optical-flow mesh refinement for `mesh` candidates, including a heavier two-stage forward/backward-validated flow pass for two-image photo seams
10. adaptive mesh sizing based on support density and residual disagreement
11. ranked candidate scoring with `rank`, `confidence`, `modelType`, and `blendMode`
12. manual fallback candidate if overlap is weak or missing

Important implementation detail:

- photo candidate traversal is currently cycle-guarded; without that, pair graphs can loop forever during anchor solving
- photo mesh refinement now supports two deformation-sharing modes:
  - `warpDistribution: 'balanced'`
    - spreads residual deformation across both images
  - `warpDistribution: 'anchored'`
    - keeps more deformation on the deeper/non-anchor side
- the photo mesh solver now seeds many more local support points from overlap detail/edge regions instead of relying only on the original inlier matches
- when exactly two photo inputs are being solved into a mesh, support seeding now deliberately favors the seam corridor instead of treating the whole overlap equally
- low-detail / blank-space regions are now deprioritized or skipped during dense support seeding so empty areas do not steer the mesh
- local mesh corrections are now thresholded twice:
  - implausibly large prepared-space optical-flow shifts are rejected
  - surviving world-space local residuals are capped to a small per-image warp budget
- two-image photo mesh refinement now uses a coarse-to-fine Lucas-Kanade pass plus forward-backward validation before residuals reach the mesh
- adaptive mesh density is now chosen per solved image from support count and residual disagreement, rather than always using one fixed grid size
- mesh residuals are intentionally regularized so unsupported outer edges stay closer to rectangular instead of letting a local seam pull the whole far edge apart
- that regularization is now less aggressive inside supported seam regions for two-image meshes, while unsupported outer regions stay more strongly pinned
- large local corrections are no longer rewarded just for being large; the mesh now prefers small believable cleanup corrections over aggressive stretches
- the Laplacian blend path now preserves combined source coverage alpha at overlap boundaries so narrow top/bottom seams do not fade out into background-colored lines
- automatic AKAZE fallback is now capped and non-fatal: Stitch keeps a usable ORB solve if the stronger retry is skipped or fails
- the Stitch settings and photo worker now recognize a `sift` feature-detector mode, and the vendored `src/vendor/opencv/opencv.js` bundle is now SIFT-enabled so explicit SIFT selection and auto escalation both work at runtime
- the repo includes `tools/generate_opencv_sift_config.py` plus `tools/build_opencv_sift.ps1` to regenerate that SIFT-enabled single-file OpenCV.js bundle
- the PowerShell build script also patches OpenCV's JS CMake input to remove the deprecated `DEMANGLE_SUPPORT` linker flag, which newer Emscripten toolchains reject during the final JS link step
- the custom OpenCV.js bundle exposes a promise-like `cv` during worker boot, so `src/stitch/opencv-worker.js` now explicitly swaps in the resolved runtime module before analysis begins; without that, ORB/AKAZE/SIFT appear missing or non-constructible in the browser worker even though the bundle contains them
- browser-side SIFT on this OpenCV.js build has a practical WASM ceiling well below many phone-photo full-res inputs, so Stitch now caps SIFT extraction to stable internal analysis sizes (roughly 2400px longest side, with smaller retries if needed), then maps those keypoints back into the original prepared-image coordinates before the rest of the photo pipeline runs

If the browser supports Workers:

- screenshot analysis runs through `src/stitch/worker.js`
- photo analysis runs through `src/stitch/opencv-worker.js`

If Worker or WASM support is missing:

- screenshot analysis can still fall back to same-thread JS analysis
- photo analysis currently requires the OpenCV worker path and will fail with a visible error instead of silently falling back

Important Stitch analysis setting:

- `settings.useFullResolutionAnalysis`
  - `true` by default for new Stitch drafts
  - when enabled, the prep step skips the normal downscale and analyzes the original image sizes
  - the screenshot search path also stays in original resolution instead of silently collapsing back to a tiny search proxy
  - this is especially useful for screenshots and UI captures, where downscaling can throw away the sharp edges and tiny features the matcher needs
  - it can be noticeably slower and use more memory

Current higher-level Stitch settings:

- `settings.sceneMode`
  - `'auto' | 'screenshot' | 'photo'`
- `settings.warpMode`
  - `'auto' | 'off' | 'perspective' | 'mesh'`
- `settings.meshDensity`
  - `'low' | 'medium' | 'high'`
- `settings.warpDistribution`
  - `'anchored' | 'balanced'`
- `settings.blendMode`
  - `'auto' | 'alpha' | 'feather' | 'seam'`
- `settings.featureDetector`
  - `'auto' | 'orb' | 'akaze'`
- `settings.photoBackend`
  - currently `'opencv-wasm'`

Current new-document defaults are intentionally photo-heavy:

- `useFullResolutionAnalysis: true`
- `warpMode: 'mesh'`
- `meshDensity: 'high'`
- `warpDistribution: 'balanced'`
- `featureDetector: 'auto'`
- `analysisMaxDimension: 2048`
- `maxFeatures: 4000`
- `ransacIterations: 5000`

Important current analysis modes in diagnostics:

- `method: 'screenshot'`
  - axis-aligned screenshot/border overlap search won
- `method: 'translation'`
  - generic translation matcher won
- `method: 'similarity'`
  - feature/RANSAC similarity matcher won
- `method: 'opencv-homography'`

Translation-oriented diagnostics can also include:

- `similarity`
- `pixelMatch`
- `edgeMatch`

Those are useful when debugging screenshot failures because they show whether the overlap was actually matching pixels or just producing a large but weak candidate.

### Stitch document and render model

Editing is world-space / resolution-free:

- images are placed in a shared coordinate system
- the editing canvas fits those bounds to the visible stage
- export rasterizes the current visible composite bounds with padding

- renderer/export/hit-test now share the same warp mesh model

- mesh candidates can now carry adaptive `cols/rows` values above the old fixed low/medium/high defaults

Candidate metadata now commonly includes:

- `rank`
- `confidence`
- `modelType`

### Stitch candidate behavior

- Analysis builds a ranked `candidates[]` list.
- After analysis, the gallery opens automatically.
- Choosing a candidate applies its placements to the current draft.
- The alternatives strip remains available so the user can switch candidates without rerunning analysis.
- screenshot-like sets tend to keep `screenshot` / `rigid` candidates on top
- photo-like sets can emit `perspective` and `mesh` candidates ahead of rigid ones
- `mesh` candidates can now share deformation across both images when `warpDistribution` is `balanced`
- photo mesh candidates try to keep unsupported outer edges closer to a sane rectangle instead of letting local overlap correction blow out the whole far edge
- non-alpha photo blends now use gain compensation plus multi-band compositing in the renderer/export path
- `blendMode: 'seam'` now uses a content-aware seam mask inside that multi-band blend instead of only a stronger rectangular feather
- `blendMode: 'auto'` keeps screenshot candidates crisp and gives photo-like candidates softer overlap-aware blends

### Important Stitch save/load behavior

- Stitch projects save to the same Library as Studio projects.
- Loading a Stitch project from the Library restores `state.stitchDocument`.
- The app automatically switches to the `Stitch` section after load.

## 8. Project Adapter System

The Library save/load/import pipeline is now engine-aware.

This is handled in `src/main.js` via `createProjectAdapterRegistry(...)`.

Current adapters:

- `studio`
- `stitch`
- `3d`

Each adapter is responsible for:

- returning the engine's current draft from app state
- determining whether that draft is empty/saveable
- serializing a clean project payload
- validating imported payloads
- rendering a Library preview blob
- restoring a payload back into the correct engine
- providing summary metadata for the Library record

This adapter layer is what makes mixed Library storage possible without hard-coding Library behavior to the Studio engine.

## 9. Library Storage Model

The Library is backed by IndexedDB:

- DB name: `ModularStudioDB`
- object store: `LibraryProjects`

There is also a reserved metadata record:

```text
id: '__library_meta__'
kind: 'library-meta'
tags: [...]
```

That record stores the persistent tag catalog so tags can exist even if no current project is using them.

### Current Library record shape

A saved project record now effectively contains:

- `id`
- `timestamp`
- `name`
- `blob`
  - rendered preview PNG blob
- `payload`
  - Studio, Stitch, or 3D project payload
- `tags`
- `projectType`
  - `'studio'`, `'stitch'`, or `'3d'`
- `hoverSource`
  - source image metadata used by Library compare hover
- `sourceWidth`
- `sourceHeight`
- `sourceAreaOverride`
  - especially important for Stitch because aggregate input area matters more than the first image size
- `sourceCount`
- `renderWidth`
- `renderHeight`

### Compare Source rule

For Studio Library projects:

- `hoverSource` is the embedded original source image.

For Stitch Library projects:

- `hoverSource` is the first uploaded input image.

This is intentional. It makes hover-compare still work in a multi-image project without inventing a composite "source".

### Source size sorting rule for Stitch

For Stitch Library projects:

- `sourceWidth/sourceHeight` track the first input image for compatibility and hover display
- `sourceAreaOverride` stores aggregate total input area across all inputs

The Library sort UI uses that override so Stitch projects sort meaningfully by original input size.

### Compare Source rule for 3D

For 3D Library projects:

- `hoverSource` is intentionally `null`
- the Library card uses the saved scene preview image instead of an original-source hover compare

```json
    "outputHeight": 1080,
    "lastJobSamples": 256
  },
  "preview": {
    "imageData": "",
    "width": 0,
    "height": 0,
    "updatedAt": 0
  }
}
```

Important 3D save/load rule:

- 3D scenes now persist embedded model data, embedded image planes, primitive items, per-item materials, texture transforms, camera presets, light target assignments, snap settings, and preview metadata directly in the payload.
- `renderJob` is intentionally excluded from serialized payloads.

### Library-export metadata fields

Single-project JSON files exported from the Library can include:

- `_libraryName`
- `_libraryTags`
- `_libraryProjectType`
- `_libraryHoverSource`
- `_librarySourceArea`
- `_librarySourceCount`

These fields are for Library round-tripping. They are stripped before normal engine validation.

## 11. Save / Load Behavior

### Save to Library

`actions.saveProjectToLibrary(...)` is now adapter-based.

Flow:

1. Determine which engine is being saved.
2. Ask that adapter for the current draft.
3. Refuse save if the draft is empty or not saveable.
4. Serialize a clean payload.
5. Fingerprint it with `makeProjectFingerprint(...)`.
6. Reuse an exact existing Library record when `preferExisting` is requested.
7. If the draft originated from a Library record, prompt for overwrite-vs-new behavior.
8. Ask the adapter to render a preview blob and summary metadata.
9. Save the resulting mixed-engine record into IndexedDB.
10. Preserve tags on overwrite.

### Load from Library

`actions.loadLibraryProject(payload, libraryId, libraryName)` now:

1. inspects the payload
2. chooses the correct adapter
3. runs the engine-specific replacement guard
4. restores the payload through that adapter
5. updates the correct active Library origin
6. switches to the correct top-level section

### Active Library origin tracking

The app no longer tracks only one active Library project.

It now tracks separate origins for:

- `studio`
- `stitch`
- `3d`

That means:

- Editor can know which Library record it came from
- Stitch can know which Library record it came from
- switching sections does not destroy overwrite context

## 12. Replacement Guards

Project replacement checks are now engine-specific.

The helper:

- `ensureProjectCanBeReplaced(projectType, actionLabel)`

does this:

1. gets the correct adapter
2. checks if that engine's current draft is empty
3. checks whether the current draft already matches an exact Library record
4. if not, asks the user whether to save first
5. if the user chooses save, it routes through `actions.saveProjectToLibrary(..., { projectType })`

This is why:

- loading a Studio state file does not look at the Stitch draft
- loading a Stitch Library project does not look at the Studio draft
- simple tab switching still does not prompt

Tag filtering supports:

The Library can save the whole Library, not just one project.

   - readable

2. Secure packed JSON

   - optional encryption

### Library import behavior

The Library import path can currently parse:

- a normal single-project JSON
- a plain Library bundle JSON
- a secure packed Library bundle JSON

If the current Library already has projects and the incoming file is a Library bundle:

- the user can merge
- or replace
- and if replacing, can optionally save the current Library first through the Save Library workflow

### Mixed-engine Library import

For each payload it:

1. parses JSON
2. extracts `_library*` metadata
3. chooses the correct adapter
4. validates the payload
5. renders a preview blob with the correct engine
6. stores a mixed-engine record in IndexedDB

This means Editor and Stitch projects can coexist in:

- single Library records
- bundle exports
- secure bundle exports
- ZIP exports

## 17. Important Gotchas

### Do not reintroduce multi-window assumptions

The current app architecture assumes:

- same page
- same runtime
- direct function calls between sections

Do not rebuild normal Library behavior around popup windows unless there is a very strong reason.

### Studio and Stitch are separate drafts

Do not collapse them into one document model casually.

Current behavior depends on:

- Studio draft surviving while the user works in Stitch
- Stitch draft surviving while the user returns to Editor or Library

### Library compare source is type-dependent

For Stitch, hover compare intentionally uses the first input image, not a synthetic composite source.

For 3D, hover compare is intentionally absent; use the saved preview render instead.

### Object URLs exist in two places

- Library gallery preview blobs
- Stitch candidate preview maps

Be careful to revoke unused blob URLs when replacing those lists.

### Old Library assumptions are now stale

These are no longer correct:

- "The app only has Editor and Library"
- "Library records are always Studio payloads"
- "The active Library project is tracked by one global id/name pair"
- "Library load always routes back into Editor"
- "The Library detail panel always shows a layer stack"

## 18. Regression Checklist After Future Changes

When touching app shell or Library logic, check:

1. `Editor <-> Stitch` tab switching does not prompt to save/discard.

## 19. Electron Bundling and Standalone App

The project now supports being bundled into a standalone Windows application (`.exe`) via Electron.

### App Configuration

- **Name**: Studio
- **Main Process**: `electron-main.js`
- **Build Tool**: `electron-builder`
- **Output**: `dist/Studio 1.0.0.exe` (Portable)

### Build Workflow

- `npm start`: Launches the Electron window for testing (faster than building).
- `npm run dist`: Compiles the project into the final `.exe`.

Current verification note:

- Packaging was re-verified on March 30, 2026 with:
  - `.\node_modules\.bin\electron-builder.cmd -w --config.directories.output=dist/verify-final-2`
  - output included `dist/verify-final-2/Studio 1.0.0.exe`
- `package.json` now excludes `dist-verify*/**/*` from Electron payloads so verification folders are not recursively bundled back into later builds.

### Mandatory Path Rule

All assets (scripts, styles, images) **MUST** be referenced using **relative paths** in `index.html` and JS files.

- **Correct**: `<script src="src/main.js"></script>`
- **Incorrect**: `<script src="/src/main.js"></script>`

This ensures the app works identically across:

1. **GitHub Pages** (served via HTTP)
2. **Local Browser** (served via Local Host or opened as a file)
3. **Electron** (loaded via the `file://` protocol)

### Security and CORS

By default, Electron runs with standard browser security. If the app needs to bypass CORS for specific external APIs, `webPreferences.webSecurity` can be disabled in `electron-main.js`. Currently, it is enabled for maximum security.

## 20. Summary

The app is no longer just "Noise Studio plus an integrated Library".

It is now:

- the original modular Studio editor
- an integrated mixed-project Library
- a second engine called Stitch
- a browser/Electron 3D scene editor with embedded model/image persistence, camera presets, material controls, and background path-traced PNG export
- a standalone Electron-powered desktop application

The key architectural shift is that the Library is now engine-aware through adapters, while the app keeps Studio, Stitch, and 3D as separate live drafts inside one page, all while maintaining universal compatibility through strict relative pathing.
