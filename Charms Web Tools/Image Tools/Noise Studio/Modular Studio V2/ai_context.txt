# Modular Studio V2 - Architecture and AI Context Guide

This file is the handoff document for future AI work on Modular Studio V2. It should describe the app as it actually works now, not as it worked in an older revision.

The current architecture is a single-page editor with an integrated Library section. The old multi-window Library model has been retired from the main workflow because it was unreliable on GitHub Pages even when it appeared fine on local hosting.

## 1. Product Model

The app has two user-facing sections inside the main site:

1. `Editor`
2. `Library`

They are switched through a top-level section bar rendered inside the main app shell. The Editor and Library now live in the same page context and no longer depend on cross-window messaging for normal use.

High-level responsibilities:

- The `Editor` owns live document state, WebGL rendering, editing controls, export, and import.
- The `Library` owns browsing saved projects, inspecting layer stacks, importing/exporting Library JSON bundles, and loading saved projects back into the Editor.

Important architectural consequence:

- The Library is now an in-app section, not a popup or secondary browser window.
- The main reason for this change was reliability on GitHub Pages.

## 2. Entry Points

### `index.html`

- Primary app entry.
- Loads `src/jszip.min.js` first so ZIP export is available to the integrated Library.
- Loads `src/main.js` as the main ES module.

### `library.html`

- No longer hosts a standalone Library app.
- It now redirects to `index.html?section=library`.
- This exists as a backward-compatibility route so old bookmarks or direct navigation still land in the Library section.

## 3. Directory Structure

### Root

- `index.html`
- `library.html`
- `style.css`
- `ai_context.txt`
- `LEGACY/`
  - Old V1-era files that are not part of the current runtime path.

### `src/`

- `src/main.js`
  - Main application entry.
  - Creates state, engine, UI, editor actions, IndexedDB helpers, and Library actions.

- `src/ui/workspaces.js`
  - Main UI shell for the single-page app.
  - Renders the top section switcher.
  - Renders the editor toolbar.
  - Renders the editor workspace shell.
  - Mounts the integrated Library panel.

- `src/ui/libraryPanel.js`
  - Integrated Library section logic.
  - Renders the Library grid/fullscreen/detail views.
  - Handles Library modals, upload/import/export, and direct calls into editor actions.

- `src/ui/previewViewport.js`
  - Preview coordinate and transform helpers.

- `src/io/documents.js`
  - JSON save/load helpers and validation.

- `src/state/store.js`
  - Lightweight store implementation.

- `src/state/documentHelpers.js`
  - View/document normalization and helper logic.

- `src/registry/`
  - Registry loading, validation, and layer instancing helpers.

- `src/engine/`
  - Custom WebGL image processing engine.

- `src/analysis/scopes.js`
  - Scope/analysis helpers for editor views.

- `src/jszip.min.js`
  - Bundled JSZip used by the Library ZIP export.

### `Shaders/`

- Physical GLSL assets used by the engine and registry.

## 4. Single-Page Shell Layout

The top-level app shell now has these main regions:

1. Section switcher bar
   - Buttons: `Editor`, `Library`

2. Editor toolbar slot
   - Visible only while the Editor section is active

3. Notice strip
   - Shared by both sections

4. Editor workspace shell
   - Visible only while `state.ui.activeSection === 'editor'`

5. Library section panel
   - Visible only while `state.ui.activeSection === 'library'`

The Editor and Library are sibling sections in the same DOM tree. They are shown/hidden by UI state, not opened in separate windows.

## 5. State Model

### Top-level store

The editor store contains:

- `state.document`
- `state.ui`
- `state.notice`
- `state.eyedropperTarget`

### `state.document`

This is the persistent/saveable document state and is still the only part the engine should care about for rendering.

Important fields include:

- `version`
- `kind`
- `mode`
- `workspace`
- `source`
- `palette`
- `layerStack`
- `selection`
- `view`
- `export`
- `batch`

### `state.ui`

This now includes the top-level section selection:

- `activeSection`
  - `'editor'` or `'library'`

It also still includes editor-only ephemeral UI flags such as:

- `compareOpen`
- `jsonCompareModalOpen`
- `jsonCompareResults`
- `jsonCompareView`
- `jsonCompareIndex`
- `loadImageOnOpen`
- `saveImageOnSave`

Important rule:

- Section switching is UI state only, so `activeSection` belongs in `state.ui`, not `state.document`.

## 6. Initial Section Routing

`src/main.js` now reads the current URL to determine the initial section:

- `?section=library` -> start in Library
- anything else -> start in Editor

`actions.setActiveSection(section)` also syncs the URL using `history.replaceState(...)` so the current section can be represented in the address bar without a full reload.

Section switching also performs some cleanup when entering the Library:

- stops batch playback
- closes compare overlays
- closes the batch dialog
- clears any armed eyedropper target

This is why `library.html` can safely redirect to `index.html?section=library`.

## 7. Editor Architecture

The Editor architecture is still fundamentally the same:

- registry-driven layers
- store-driven UI
- custom WebGL engine
- preview and export pipeline

Flow:

1. UI interaction calls an `actions.*` method.
2. The store updates.
3. `createWorkspaceUI(...).render(state)` rerenders the DOM shell.
4. If the update requests rendering and the engine has an image, the engine rerenders using `state.document`.

The section integration did not change the core render engine model.

## 8. Layer System

Layers are still defined primarily in `src/registry/layerRegistry.json`.

Each layer definition includes:

- `layerId`
- group metadata
- defaults
- UI controls
- bindings
- executor/shader information

The current runtime still uses registry helpers for:

- layer instancing
- validation
- relabeling
- normalization

`reindexStack(...)` remains important because it rebuilds normalized layer instances from registry defaults and persisted params.

## 9. Library System - New Architecture

The Library is now an integrated section implemented by `src/ui/libraryPanel.js`.

This file replaces the old standalone `src/ui/library.js` workflow.

### 9.1 Core design

The Library does not maintain its own copy of live editor state.

Instead it works from IndexedDB records that contain:

- `id`
- `timestamp`
- `name`
- `blob`
- `payload`

The Library panel reads from IndexedDB and renders:

- grid view
- fullscreen click-through view
- project detail overlay
- modal confirmations/alerts
- upload/render progress overlay

### 9.2 Why the old window model was retired

The previous system used a separate `library.html` page and cross-window coordination. Even after attempts to harden the transport, the GitHub Pages deployment still failed in real use while local hosting appeared fine.

The underlying issue was not the Library data model itself. It was the browser-context boundary:

- separate windows
- popup/window linkage
- refresh/reopen state
- hosting-specific behavior

The single-page Library eliminates that entire failure class by making Library actions direct function calls inside one page.

## 10. IndexedDB Library Storage

Database:

- `ModularStudioDB`

Object store:

- `LibraryProjects`

Key path:

- `id`

Current record shape:

```js
{
  id: String,
  timestamp: Number,
  name: String,
  blob: Blob,
  payload: Object
}
```

Editor-side helper functions in `src/main.js`:

- `initDB()`
- `saveToLibraryDB(project)`
- `deleteFromLibraryDB(id)`
- `clearAllFromLibraryDB()`
- `getAllFromLibraryDB()`

These are still kept in `main.js` because the Editor owns the authoritative Library persistence actions.

## 11. Save-to-Library Logic

`actions.saveProjectToLibrary(nameOverride, optionsOrForceNew)` is the main persistence entry point.

Behavior:

1. Requires a loaded source image with `source.imageData`.
2. Builds a full document-style payload, not just layer data.
3. Current Library payload shape is effectively:

```js
{
  version: 'mns/v2',
  kind: 'document',
  mode: 'studio',
  workspace,
  palette,
  selection,
  view,
  export,
  source,
  layerStack
}
```

4. Generates a fingerprint with `makeProjectFingerprint(payload)`.
5. If the caller uses `preferExisting: true`, an identical existing Library record is reused instead of creating another copy.
6. If the currently active Library record is unchanged, it can short-circuit with "Already saved - no changes detected."
7. When editing a project originally loaded from the Library, it prompts for overwrite-vs-new behavior.
8. When the current project already matches the active Library record, the prompt wording changes so the user can either keep the current saved version or type a new name to branch a manual version.
9. Renders a fresh preview PNG blob through the engine.
10. Saves the record into IndexedDB.
11. Updates `activeLibraryId` / `activeLibraryName`.
12. Calls `notifyLibraryChanged()` so the integrated Library panel refreshes.

Important nuance:

- The app no longer deletes duplicate records globally during save.
- Manual saves can preserve intentionally named versions.
- Protective/automatic save flows prefer reusing an exact existing match.

## 12. Tracking the Active Library Record

The editor still tracks the currently loaded Library entry with:

- `activeLibraryId`
- `activeLibraryName`

This matters because it controls whether saving should:

- overwrite the current Library entry
- or create a new one

The integrated Library also uses this indirectly when loading records back into the Editor.

Important lifecycle rule:

- `activeLibraryId` / `activeLibraryName` are cleared when an external file import starts a new branch of work.
- This now happens for `newProject()`, `openImageFile(...)`, and successful `openStateFile(...)` imports.
- This prevents a later save from accidentally overwriting an older Library entry that no longer represents the current editor state.

## 13. Direct Library Actions in `main.js`

The integrated Library now calls these direct actions instead of sending window messages:

- `actions.getLibraryProjects()`
  - Reads all Library records.

- `actions.deleteLibraryProject(id)`
  - Deletes a single Library record.
  - Clears `activeLibraryId` / `activeLibraryName` if the deleted record was currently active.
  - Refreshes the integrated Library.

- `actions.clearLibraryProjects()`
  - Clears the entire Library store.
  - Resets `activeLibraryId` / `activeLibraryName`.
  - Refreshes the integrated Library.

- `actions.loadLibraryProject(payload, libraryId, libraryName)`
  - Directly loads a saved Library payload into the editor.
  - Calls a shared replacement guard before swapping editor state.
  - If the current project is not already backed by an exact Library match, the user is prompted to save it or explicitly discard it.
  - Accepts both current `mns/v2` payloads and older legacy `version: 2` payloads.
  - Loads embedded source image data into the engine if present.
  - Reindexes the layer stack.
  - Restores active Library origin tracking.
  - Switches back to the `Editor` section.

- `actions.processLibraryPayloads(payloadsText, filenames, onProgress)`
  - Takes uploaded Library JSON payloads.
  - Normalizes legacy `version: 2` payloads into `mns/v2` before validation.
  - Validates them.
  - Uses the editor engine to render preview blobs.
  - Saves new Library records.
  - Reports progress through a direct callback instead of cross-window messages.
  - Restores the previous source image in the engine afterward when possible.
  - Explicitly rerenders the current editor document after engine restoration when the editor still has a source image.
  - Refreshes the integrated Library.

- `actions.openLibrary()`
  - Now just switches the app to the `Library` section.
  - It no longer opens a popup or a secondary window.

## 13.1 Replacement Safeguards

The current app now treats several actions as "project replacement" or "project branching" moments:

- `newProject()`
- `openImageFile(file)`
- `openStateFile(file)`
- `loadLibraryProject(...)`

They now share a common rule:

1. If there is no current source image, proceed immediately.
2. If the current editor state already exactly matches something in the Library, proceed without another save prompt.
3. Otherwise ask whether to save the current project to the Library first.
4. If the user declines saving, ask for explicit discard confirmation before replacing state.

This is the main guard against orphaning in-progress work during common user journeys.

## 14. Integrated Library Panel Behavior

`src/ui/libraryPanel.js` owns all Library-specific UI logic.

### 14.1 Internal state

It keeps local UI state for:

- `libraryData`
- hover-original mode
- fullscreen/grid mode
- current fullscreen index
- currently open detail record
- refresh token to avoid stale async refresh results

This state is intentionally local to the Library panel because it is UI-only and should not pollute the editor store.

### 14.2 Data source

The Library panel does not mirror the Library into the global store.

Instead it calls:

- `actions.getLibraryProjects()`

and transforms records into UI items with:

- `url` from `URL.createObjectURL(blob)`
- `hoverSrc` from `payload.source.imageData`

### 14.3 Refresh model

The integrated Library refreshes through direct calls, not message passing.

Main trigger points:

- activating the `Library` section
- saving to the Library
- deleting a Library record
- clearing the Library
- processing uploaded Library JSON files
- explicit `view.refreshLibrary()` calls from `main.js`

### 14.4 Detail modal defaults filtering

The old standalone Library used a runtime handshake to obtain layer defaults.

That handshake is gone.

The integrated Library now gets defaults directly from the loaded registry via `workspaces.js`, which passes:

- `layerId -> defaults`

into `createLibraryPanel(...)`.

This is more reliable because it removes yet another communication boundary.

## 15. Library UI Features Preserved

The integrated Library preserves the major behaviors of the old Library:

- grid browsing
- hover-to-show-original
- fullscreen click-through browsing
- detail modal
- load into editor
- delete single project
- clear all with confirmation
- save full library as one JSON bundle
- export all projects as zipped individual JSON files
- upload JSON / bulk render into Library

Notable improvement:

- All of this now happens without relying on another page being open.

## 16. Upload / Render-to-Library Flow

When the user uploads JSON from the integrated Library:

1. `libraryPanel.js` opens a file input.
2. It reads the selected files as text.
3. It unrolls bundled arrays using `_libraryName` when present.
4. It calls `actions.processLibraryPayloads(unrolledPayloads, unrolledNames, setProgress)`.
5. The editor engine normalizes older legacy payloads if needed, validates them, renders preview blobs, and stores them in IndexedDB.
6. The Library panel refreshes directly after completion.

The progress overlay is now driven by the direct `onProgress` callback instead of a transport message like `START_RENDER` or `UPDATE_PROGRESS`.

## 17. ZIP Export

The integrated Library uses the global `window.JSZip` loaded from `src/jszip.min.js` in `index.html`.

ZIP export behavior:

- one `.mns.json` file per Library record
- payloads are normalized to current `mns/v2` document shape during export
- filename preserved where possible
- downloaded as `modular_studio_exports.zip`

## 18. Redirected `library.html`

`library.html` is now just a redirect page.

Purpose:

- preserve older links/bookmarks
- provide a direct path into the Library section

It should not be treated as an independent runtime surface anymore.

## 19. Removed/Retired Architecture

These are no longer part of the primary architecture:

- standalone `src/ui/library.js`
- popup-based `window.open('library.html', ...)` Library flow
- cross-window transport abstractions for Library/editor communication
- Library default handshake messages
- Library refresh via browser window messaging

If these concepts appear in older notes, they are outdated.

## 20. Hosting Notes

The original reason for the integrated Library refactor was deployment reliability on GitHub Pages.

Current expectation:

- Since the Library is in the same page, Pages should no longer depend on popup behavior or cross-window state.
- The app still depends on browser features like IndexedDB, File APIs, and WebGL, but not on a second window or tab for Library use.

If a future GitHub Pages issue appears, investigate in this order:

1. IndexedDB behavior
2. path resolution / asset loading
3. source image embedding in payloads
4. WebGL/rendering behavior
5. section-routing / URL-state behavior

Do not assume the problem is still a cross-window issue, because that architecture is no longer the primary path.
